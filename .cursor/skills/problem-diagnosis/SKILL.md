# 问题诊断排查

基于《个人赚钱成长操作系统》第2篇第7章「问题处理」。详细原则见 [reference.md](reference.md)。

## 核心原则

> 「问题不在于会不会卡。问题在于：你卡住时有没有一套处理机制。」
> 「事实先于情绪。」

## 诊断流程

### 第1步：事实描述

引导用户只写现象，不写情绪：
- 当前卡点现象是什么？
- 具体发生了什么事实？
- 不要写"我觉得不行了"，要写"发了5条内容，0咨询"。

### 第2步：六层逐层排查

按以下顺序逐层排查，每层问具体问题：

**第1层 项目层**：
- 需求是不是真实存在？
- 用户有没有支付意愿？
- 这个方向是不是只是热闹？
- 项目成立性是否不足？

**第2层 流量层**：
- 平台是否选对？
- 用户是否会出现在这里？
- 表达是否击中需求？
- 流量入口是否太弱？

**第3层 转化层**：
- 用户是否理解价值？
- 产品/服务描述是否清楚？
- 信任是否不足？
- 定价是否错位？
- 成交动作是否太复杂？

**第4层 交付层**：
- 交付是否太重？
- 交付边界是否不清？
- 是否过度依赖个人状态？
- 是否容易翻车？

**第5层 执行层**：
- 动作是否太少？
- 动作是否过散？
- 是否一直没进入真实测试？
- 执行是否经常断线？

**第6层 认知偏差层**：
- 是否高估了这个项目？
- 是否把表面形式当本质？
- 是否误判了用户需求？
- 是否把偶然当稳定？

### 第3步：定位核心问题层

排查完后，明确告诉用户：「你当前最核心的问题在第X层——XX层。」只定位一个主问题层。

### 第4步：单变量调整

只改一个关键变量，不要同时调整多个：
- 这轮只调整一个什么变量？
- 为什么先调这个？
- 调整后怎么重新验证？

### 第5步：设计下轮验证

- 下一轮准备怎么重新验证？
- 本轮不做什么（明确边界）？

## 输出格式

读取 `.cursor/模板/模板9：项目问题排查表.md` 的结构作为输出模板。

## 完成后衔接

- 如果问题在项目层 → 建议回到分析阶段重新评估，说「帮我分析」
- 如果问题在执行层 → 建议重新拆解，说「帮我设计闭环」
- 如果调整后跑了新一轮 → 建议说「帮我复盘」
