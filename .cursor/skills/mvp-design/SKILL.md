# 最小闭环设计与执行拆解

基于《个人赚钱成长操作系统》第2篇第4-6章。详细原则、案例见 [reference.md](reference.md)。

## 核心原则

> 「最小闭环的本质就是：用最小成本，把想法推进到真实反馈。」
> 「你前期真正需要的，不是一个完美项目。而是：一个最小产品、一个最小入口、一个最小成交动作、一个最小反馈机制。」

## 设计流程（4步）

### 第1步：定义验证目标

引导用户明确本轮最小闭环要验证什么。常见验证目标：
- 有没有真实需求
- 有没有人愿意咨询
- 有没有人愿意付费
- 我能不能初步交付
- 某个流量入口有没有反应

提问：「你这一轮最想验证的一件事是什么？」验证目标越清晰，MVP越不会乱。

### 第2步：把完整项目砍到最小

引导用户回答：
- 这个项目如果只留10%的核心，先做什么？
- 如果只能先交付最基础价值，给什么？
- 如果只能先接一个人，怎么做？

最小产品形态参考：一份小资料、一个简版服务、一个基础版模板、一个小范围咨询、一个简化版工具页。

### 第3步：明确流量入口

一次只挑一个最可能触达目标用户的入口：
- 一条测试内容 / 一个朋友圈表达 / 一个群内试探 / 一个搜索承接页 / 一个垂类社区帖子

提问：「你最容易在哪里接触到你的目标用户？」

### 第4步：定义反馈标准

必须提前定义，否则容易陷入自我解释：
- 什么算有反馈
- 什么算无反馈
- 多少咨询算值得继续
- 什么样的冷淡说明可能有问题
- 多久后进行判断

## 常见错误检查

设计完后检查用户是否犯了以下错误：
1. MVP做得一点都不小（还在追求完整感）
2. 没有明确验证目标（不知道在验证什么）
3. 没有真实接触用户（只是自己写了放那里）
4. 一次想测太多变量（需求+平台+定价+包装全想测）

如果发现，直接指出并帮助修正。

## 执行拆解（4层）

MVP设计完后，立刻帮用户做执行拆解：

**第1层 目标**：本阶段要达到什么可观察状态（如"拿到3个真实咨询"）

**第2层 阶段**：当前处于哪一段（收集与理解/初筛与确定/最小闭环设计/流量测试/成交尝试/初步交付/复盘优化）

**第3层 动作**：具体要做什么，足够小、足够清楚（如"整理10个同类案例"、"发一条测试内容"、"私聊3个潜在用户"）

**第4层 节点判断**：做到什么程度该继续/该调整/该停（如"发出3条内容完全无反馈→回头看表达和需求"）

四条拆解原则：
1. 不做大而空目标，只做当前可执行动作
2. 动作尽量靠近反馈
3. 动作数量不要过多
4. 每天都要知道「今天最重要的一步是什么」

## 时间安排建议

基于五个稳定执行原则给出建议：
1. **低摩擦启动**：先做15分钟也行，先只做一个动作也行
2. **固定时段**：固定一个相对稳定的时段
3. **固定动作**：把时段和动作做一定绑定
4. **低预期坚持**：接受现实节奏、接受慢一点、重点是不断线
5. **形成节奏感**：知道什么时候输入、输出、测试、复盘、休整
